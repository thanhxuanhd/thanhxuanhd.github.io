CRUD Operations Command Center

This section represents the bulk of your exam score. Navigate through the interactive tabs to explore the nuances of creating, reading, updating, and aggregating data, including identifying common syntax pitfalls designed to trick you on the test.

✔ Properly Formed

insertOne requires a single object. insertMany requires an array of objects.

// Single Document db.users.insertOne({ "name": "Alice", "role": "admin" }) // Multiple Documents (Note the array brackets) db.users.insertMany([ { "name": "Bob", "age": 25 }, { "name": "Charlie", "age": 28 } ])

✘ Improperly Formed (Traps)

Watch out for missing arrays, multiple arguments to single inserts, or non-string keys.

// ERROR: Passing multiple args instead of array db.users.insertOne({ "name": "Alice" }, { "name": "Bob" }) // ERROR: Missing array [] inside insertMany db.users.insertMany({ "name": "Bob" }, { "name": "Charlie" }) // ERROR: Key is an integer, must be a string db.users.insertOne({ 123: "Invalid Key Type" })
DANGER

Full Document Replacement

Executing an update operation without using update operators (like $set) replaces the entire document, permanently losing unspecified fields.

db.users.updateOne( { "_id": 101 }, { "name": "Alice", "status": "Active" } )
SAFE

Targeted Update

The $set operator modifies only specified fields, leaving the rest of the document completely intact.

db.users.updateOne( { "_id": 101 }, { $set: { "city": "Chicago" } } )

Upsert & Atomicity Dynamics

The Upsert Pattern

upsert: true updates if found, inserts if missing. $setOnInsert is ignored unless a new document is created.

db.users.updateOne( { "email": "new@example.com" }, { $set: { "role": "member" }, $setOnInsert: { "joinedAt": new Date() } }, { upsert: true } )

Atomic Concurrency

findAndModify performs finding and updating in one atomic step, preventing race conditions where multiple threads might fetch the same document state.

If Thread A runs findAndModify to change status from "pending" to "processing", Thread B attempting an updateOne on "pending" will instantly find 0 matches, safely avoiding a collision.

Array Querying

MongoDB behaves differently based on operators when targeting arrays.

// 1. Array Inclusion (Any match) db.coll.find({ "tags": { $in: ["green", "blue"] } })
// 2. Specific Element Criteria // Matches if one array element satisfies ALL conditions db.orders.find({ "items": { $elemMatch: { "product": "banana", "qty": { $gt: 10 } } } })

Cursor Handling

  • toArray(): Converts all docs into a memory array. Memory-intensive!
  • forEach(cb): Iteratively streams elements to save memory.
  • countDocuments(): Scans index/docs for accurate counts based on a query filter.

Sorting, Limiting & Projections

Sort order (`1` ascending, `-1` descending) is always applied before Limits.

Projection Error Trap ⚠️

You cannot mix inclusions (`1`) and exclusions (`0`) in a single projection, with the sole exception of the _id field.

✘ db.users.find({}, { "name": 1, "age": 0 })
✔ db.users.find({}, { "name": 1, "_id": 0 })

The Pipeline Approach

Aggregations process data records and return computed results. Documents pass through stages sequentially, transforming at each step.

1

$match

Filters the document stream early to reduce processing load. Syntax is identical to standard queries.

{ $match: { "status": "completed" } }
2

$group

Groups documents by a specified _id key and applies accumulator operations (e.g., $sum).

{ $group: { "_id": "$product", "rev": { $sum: "$amount" } } }
3

$lookup / $out

$lookup joins collections. $out persists results to a new collection, overwriting if it exists.

{ $out: "revenue_report" }

Indexes & Data Modeling

Without indexes, MongoDB must scan every document to fulfill a query. This section compares the performance impact of indexing and highlights structural anti-patterns that degrade database efficiency.

Performance: IXSCAN vs COLLSCAN

Theoretical execution time for a query matching 50 documents out of a 1,000,000 document collection.

Resolving In-Memory Sorts

If explain() reveals a SORT stage instead of an index scan, create a compound index matching the exact sort direction to eliminate RAM overhead:
db.coll.createIndex({ "lastName": 1, "firstName": -1 })

Embedding vs. Linking Strategy

⊂ Embed (Denormalize)

  • • 1:1 or 1:Few relationships
  • • Data rarely changes
  • • Elements accessed together

∞ Link (Reference)

  • • 1:Many (unbounded growth)
  • • Many-to-Many relationships
  • • Frequently updated records

Data Modeling Anti-Patterns

  • 1

    Unbounded Array Growth

    Storing unlimited comments inside a post array will eventually breach the strict 16MB BSON document limit, causing app crashes.

  • 2

    Over-Referencing Static Data

    Creating separate collections for tiny, static data points (like country codes) causes inefficient, unnecessary lookup joins.

Drivers & Connection Architecture

Understanding how application drivers (like C#/.NET) communicate with the cluster is crucial. This covers connection strings, synchronous vs asynchronous operations, and the efficiency mechanics of connection pooling.

C# Driver Operations

The C# driver heavily utilizes Builders for type-safe query construction.

// Update Many Example var filter = Builders<Product> .Filter.Eq(p => p.Name, "Chair"); var update = Builders<Product> .Update.Set(p => p.Price, 50); await collection .UpdateManyAsync(filter, update);
// Find Async Example var products = await collection .Find(p => p.Price > 50) .ToListAsync();
